# 에피소드 검색 전략 비교 실험 제안

## 배경

현재 아름의 검색 방식은 **태그 겹침 점수** 기반이다.
사용자 입력에서 감정/상황 태그를 추출하고, 구절별 태그와 겹치는 수를 세어 정렬한다.

이 방식의 핵심 한계:

- **태깅 품질 의존도**: 태그가 잘못 붙어 있으면 좋은 에피소드가 누락됨
- **Vocabulary mismatch**: 사용자가 "지쳐요"라고 하면 "피로" 태그와 매칭되지만, "번아웃", "기력이 없어요" 같은 표현은 놓칠 수 있음
- **에피소드 DB 미연결**: `data/episodes/`에 2-step으로 축적된 에피소드 텍스트가 검색에 활용되지 않고 있음

---

## 검토한 접근법

### 1. BM25 (Sparse Retrieval)

에피소드 텍스트(제목 + 1-step + 2-step) 전체를 단어 단위로 인덱싱하고,
BM25 알고리즘으로 쿼리와 문서 간 단어 겹침 점수를 계산한다.

| 항목 | 내용 |
|------|------|
| 인프라 | `rank-bm25` 라이브러리 (pip, 경량) |
| 장점 | 빠름, 저렴, 결과 예측 가능, 디버깅 쉬움 |
| 단점 | 동의어/우회 표현 취약 ("지쳐요" vs "소진됨") |
| 특이사항 | 2025 벤치마크에서 도메인 특화 텍스트(금융, 법률)에서 상업 임베딩 모델보다 강한 사례 있음. 성경 텍스트도 신학 용어가 많아 유사한 특성일 수 있음 |

---

### 2. Embedding (Dense Retrieval)

에피소드 텍스트를 벡터로 변환해 저장하고,
쿼리 벡터와의 코사인 유사도로 검색한다.

| 항목 | 내용 |
|------|------|
| 인프라 | OpenAI Embeddings API |
| 장점 | 의미적 유사도 포착, 동의어/우회 표현에 강함 |
| 단점 | 임베딩 모델 필요 (비용 또는 로컬 모델), 왜 매칭됐는지 설명 어려움 |
| 특이사항 | 범용 임베딩 모델은 신학 도메인에 약할 수 있음. 한국어 지원 모델 선택 필요 |

---

### 3. Hybrid (BM25 + Embedding)

두 점수를 정규화 후 가중 합산한다.
`score = α × BM25 + (1-α) × Embedding`

| 항목 | 내용 |
|------|------|
| 인프라 | BM25 + Embedding 모두 필요 |
| 장점 | 두 방식의 장점 결합. 2025 학계 표준 패턴 |
| 단점 | α 값 튜닝 필요 |
| 참고 | LiveRAG Challenge 2025에서 Hybrid가 단일 전략 대비 일관되게 우세 |

---

### 4. BM25 + LLM Reranking

BM25로 후보 20개를 빠르게 추출하고,
LLM에게 "이 중 사용자 고민과 가장 관련 있는 것을 순서대로 골라라"를 요청한다.

| 항목 | 내용 |
|------|------|
| 인프라 | BM25 + Claude API (Haiku) |
| 장점 | LLM의 문맥 이해력 활용. 정확도 높음 |
| 단점 | 쿼리당 API 호출 추가 발생 (비용, 지연) |
| 참고 | LiveRAG 2025에서 MAP 0.523 → 0.797 (52% 향상), 단 응답 시간 48배 증가 |

## 실험 설계

이론적으로 우열을 판단하기 어렵기 때문에 **실제 쿼리로 비교 실험**이 필요하다.

### 실험 구성

```
입력:  테스트 쿼리 20~30개 (실제 기도제목/고민 문장)
출력:  쿼리별 상위 결과 나열 (수동 검토용)
```

### 평가 지표

정성평가 수동 검토

### 테스트 쿼리 예시

```
"회사에서 믿었던 동료한테 배신당했어요."
"기도를 해도 응답이 없는 것 같아요."
"용서하기가 너무 힘들어요."
"두렵고 앞이 안 보여요. 어떻게 해야 할지 모르겠어요."
"돈 걱정이 너무 심해요. 생활이 너무 힘들어요."
"제가 너무 죄인 같아서 하나님 앞에 나아갈 자격이 없는 것 같아요."
"아무도 저를 알아주지 않는 것 같아요."
"사람들 앞에서 창피를 당했어요."
"가족과의 관계가 너무 힘들어요."
"하나님이 정말 존재하시는지 모르겠어요."
```

## 구현 범위 (스크립트 1개)

```
evals/eval_retrieval.py
  - 에피소드 로딩 (data/episodes/ 전체)
  - 4가지 전략 구현 (BM25, Embed, Hybrid, LLM)
  - 임베딩 캐시 (재실행 시 재계산 불필요)
  - CLI 옵션: --strategies, --top-k, --alpha, --llm-pool
  - 출력: 쿼리별 결과

evals/eval_queries.json
  - 테스트 쿼리 + 쿼리별 결과
```

의존성:
```
rank-bm25                  # BM25
OpenAI text-embedding-3-small      # Embedding (API)
anthropic                  # LLM Reranking (이미 설치됨)
```

---

## 권장 순서

1. **1단계**: BM25 단독으로 먼저 실험 → 현재 태그 방식 대비 얼마나 좋은지 확인
2. **2단계**: Embedding 추가 → Hybrid 조합 비교
3. **3단계**: 결과가 만족스럽지 않을 때만 LLM Reranking 도입 (비용/속도 trade-off)

ElasticSearch 도입은 프로덕션 단계에서 고려. 현재 852개 에피소드 규모에서는 경량 라이브러리로 충분하다.
