# 아름 (Areum) — 프로젝트 기획서 v0.4

**작성일**: 2026-05-27
**버전**: v0.4 (DB 범위 신약 전체로 확장, 2-step 처리 방식 확정)

---

## 1. 프로젝트 개요

**아름**은 기독교인을 위한 말씀 묵상 + 기도 관리 AI 앱이다.

사용자가 오늘의 고민이나 기도제목을 입력하면, AI가 성경의 맥락을 이해한 채로 관련 에피소드를 연결해준다. 그리고 쌓인 기도들이 시간이 지나면서 자기 자신을 비추는 거울이 된다.

---

## 2. 이름의 뜻

**아름**은 세 가지 의미를 동시에 품는다.

| 의미 | 설명 |
|------|------|
| 아 (我) | 나, 자아 |
| 아름 (embrace) | 두 팔로 감싸 안을 수 있는 분량 — 포용, 품음 |
| 앎 (knowing) | 알아가는 것 |

이름 자체가 제품 철학이다: *나를 알아가는 것, 그것을 품는 것.*

---

## 3. 핵심 철학

> **AI는 방향을 주는 게 아니라 비춰주는 역할이다.**

대부분의 AI 도구는 지시적(directive)이다 — 무엇을 해야 하는지 말해준다. 아름은 반대다. AI의 역할은 사용자 안에 이미 있는 것을 표면으로 끌어올리는 것이지, 새로운 무언가를 주입하는 것이 아니다.

이 원칙은 기독교 맥락에서 더 중요해진다. AI가 예수님의 목소리를 대변하거나 신앙의 방향을 지시해서는 안 된다. 성경 에피소드를 연결할 뿐이다 — 해석의 주체는 사용자 본인이다.

---

## 4. 문제 정의

기독교인들은 말씀과 기도를 실천하고 싶어 한다. 하지만 현실은:

- **말씀**: 오늘의 고민과 관련된 구절을 스스로 찾기 어렵다. 성경은 방대하고, 연결은 직관에 의존한다.
- **기도**: 매일 기도하지만 내가 무엇을 기도했는지 기억하지 못한다. 기도가 쌓이지 않는다.
- **묵상**: 구절을 읽어도 내 삶의 맥락과 연결되는 언어가 없다.

아름은 이 세 가지 gap을 동시에 메운다.

---

## 5. 제품 구조

### 두 개의 축

**축 1: 말씀 관리**
- 성경 전체를 에피소드 단위로 정리한 DB 구축
- 사용자가 고민/기도제목을 입력하면 관련 에피소드를 연결
- 에피소드의 신학적 맥락과 고민 사이의 연결을 언어로 명시

**축 2: 고민/기도 축적**
- 사용자의 기도제목을 저장하고 쌓음
- 시간이 지날수록 자신이 기도했던 내용을 돌아볼 수 있음
- **축적이 곧 가치** — 내용이 쌓일수록 자신을 알아가는 구조

## 6. 성경 DB 설계: 에피소드 단위

### 왜 절(verse) 단위가 아닌가

성경의 의미 단위는 절이 아니라 단락(페리코페, pericope)이다. 절을 맥락 없이 분리하면 표면적 단어만 남아 전혀 다른 의미로 매칭될 수 있다.

**문제 사례**: 누가복음 7:45
> "당신은 내 얼굴에도 입을 맞추지 않았지만, 이 여자는 들어오면서부터 거듭해서 내 발에 입을 맞추었소."

- 절만 보면: "이해", "사랑", "반복적 행동" → 가족 관계 기도에 매칭될 수 있음
- 에피소드(7:36-50) 맥락: 외식하는 바리새인 vs. 진심으로 회개하는 여인의 대조 → 죄 사함에서 나오는 감사와 사랑

맥락을 모르면 연결이 아니라 오독이 된다.

### 에피소드 DB 구조

**범위: 신약 전체 (마태복음 ~ 요한계시록)**

새번역 성경 소제목을 에피소드 경계로 사용한다. 신약 27권, 총 852개 에피소드.

```
에피소드 페이지 1개:
  - 범위: 누가복음 7:36-50
  - 제목: 향유를 부은 여인
  - 1-step (상황): 바리새인 시몬이 예수님을 초대했으나 관습적 환대를 하지 않음.
                   죄인이라 불리던 여인이 눈물로 발을 씻기고 향유를 부음.
  - 2-step (의미): 외식하는 신앙(시몬) vs. 진심 어린 회개(여인)의 대조.
                   죄를 많이 사함 받은 자가 더 큰 사랑을 나타냄.
                   진정한 감사는 형식이 아닌 마음에서 나온다.
  - 원문: 핵심 절 포함
```

### 2-step 처리 방식

**1-step (상황 정리)**: AI가 에피소드의 등장인물, 사건, 배경을 요약한다.
- 무엇이 일어났는가
- 누가, 왜, 어떤 상황에서

**2-step (의미 해석)**: 정리된 상황을 바탕으로 신학적·감정적 의미를 추출한다.
- 이 에피소드가 말하는 핵심은 무엇인가
- 어떤 사람의 어떤 상황에 닿을 수 있는가

두 단계를 분리하는 이유: 상황 요약 품질과 의미 해석 품질을 각각 독립적으로 평가하고 개선할 수 있어야 한다.

## 7. AI 레이어 설계: 두 가지 번역

### 번역 1: 사람 언어 → 에피소드 매칭

사용자의 고민을 에피소드 DB의 의미 요약과 비교하여 가장 관련 있는 에피소드를 찾는다.

```
"직장에서 인정 못 받는 것 같아"
→ 에피소드 요약들과 의미적 비교
→ 마태복음 20:1-16 (포도원 품꾼의 비유) — 인간의 공정함 기대 vs. 하나님의 은혜
```

태그 매칭이 아니라 LLM이 에피소드 요약을 직접 읽고 의미적 연관성을 판단한다.
에피소드 수(500-800개)가 LLM이 인덱스를 읽고 후보를 좁힌 뒤 직접 비교하기에 충분한 규모다.

### 번역 2: 에피소드 → 사람 언어

선택된 에피소드를 사용자의 고민 맥락에 맞게 따뜻한 언어로 전달한다.

```
마태복음 20:1-16
→ "인정받지 못한다는 느낌이 드시는군요.
   이 말씀이 생각났어요. 예수님은 세상의 기준과 다른 방식으로 각 사람을 대하세요..."
```

원문 + 맥락 요약 + 고민과의 연결이 함께 제공된다.

### 태그에 대하여

현재는 명시적 태그를 사용하지 않는다.

태그는 의미를 키워드로 압축한 대리물(proxy)이다. 에피소드 의미 요약이 있으면 대리물보다 원본이 더 정확하다. 태그는 나중에 그래프 시각화나 대규모 검색이 필요할 때 추가한다.

## 8. 주요 설계 결정

| 결정 | 선택 | 이유 |
|------|------|------|
| 성경 DB 범위 | 신약 전체 (852 에피소드) | 구약 포함 시 규모 과대. 신약으로 먼저 완성 후 확장 |
| 성경 DB 단위 | 에피소드 (페리코페) | 절 단위는 맥락 손실로 오매칭 발생 |
| 에피소드 경계 | 새번역 소제목 기준 | 신학적으로 검증된 단위. 직접 분류 불필요 |
| DB 처리 방식 | 2-step (상황 → 의미) | 품질 평가·개선을 단계별로 독립 수행 가능 |
| 매칭 방식 | LLM 의미 비교 | 에피소드 요약이 있으면 태그보다 정확 |
| 명시적 태그 | 사용 안 함 (우선) | 시각화 필요 시 추후 추가 |
| 묵상 콘텐츠 | 성경 원문 | 저작권 있는 묵상집 사용 불가. 파트너십은 추후 |
| 주석 포함 여부 | 제외 (우선) | 규모 가늠 안 됨. 나중에 추가 가능 |
| 번역본 | 미결 (라이선스 확인 필요) | 개역개정은 대한성서공회 저작권 |

## 9. MVP 범위

**터미널 UI (v0.1 MVP)**

- 성경 DB 범위: 신약 전체 (마태복음 ~ 요한계시록), 2-step 처리 완료 상태

1. 사용자가 오늘의 고민/기도제목 입력
2. LLM이 에피소드 DB 인덱스를 읽고 후보 에피소드 선택
3. 선택된 에피소드의 맥락과 의미를 고민에 연결하여 전달
4. 기도 내용 저장 (로컬 마크다운)

**범위 밖 (추후)**
- 웹/앱 UI
- 임베딩 기반 검색
- 태그 및 그래프 시각화
- 묵상 파트너십 콘텐츠

## 12. 핵심 리스크

**"AI가 성경을 해석한다"는 오해**
아름은 해석하지 않는다 — 연결할 뿐이다. 이 경계를 UX 언어와 온보딩에서 명확하게 전달해야 한다.

**번역본 라이선스**
개역개정(대한성서공회)은 상업적 사용 시 별도 허가 필요. 공개 번역본 검토 필요.

**에피소드 DB 구축 비용**
성경 전체를 에피소드 단위로 나누고 2-step 처리하는 작업은 한 번의 선투자다. 품질이 매칭 정확도를 결정하므로 이 단계를 허술하게 넘기면 안 된다.

**축적 가치의 지연**
앱의 핵심 가치는 쌓일수록 드러난다. 초기 사용자 경험(1회차)이 빈약하면 이탈한다. 첫 세션부터 가치가 느껴지도록 설계해야 한다.
